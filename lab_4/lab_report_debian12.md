## Часть 1. Что я сделала и зачем

### Шаг 0 — Подготовка виртуальных машин в UTM

Создала три виртуальные машины с Debian/Ubuntu:

| VM | Роль | Сетевые адаптеры |
|----|------|-----------------|
| VM1 | Маршрутизатор / NAT / Firewall | enp0s1 (Bridged/Shared → интернет) + enp0s2 (Host Only → внутренняя сеть) |
| VM2 | Клиент | enp0s1 (Host Only → внутренняя сеть) |
| VM3 | Клиент | enp0s1 (Host Only → внутренняя сеть) |

**Почему два адаптера у VM1?**
VM1 должна стоять «между» интернетом и внутренней сетью. Один интерфейс смотрит наружу, другой — внутрь. Именно через неё весь трафик VM2/VM3 уходит в интернет.

**Важный нюанс UTM:** в UTM нет режима «Internal Network» как в VirtualBox. Использовался режим **Host Only** для внутренней сети. Галочка «Изолировать гостя от хоста» должна быть **снята** — иначе VM не видят друг друга.

### Шаг 1 — Определение имён интерфейсов

```bash
ip a
```

**Зачем:** в Debian/Ubuntu имена интерфейсов зависят от оборудования. В UTM оказались `enp0s1` и `enp0s2` вместо `enp0s3`/`enp0s8` из методички. Все дальнейшие команды использовали реальные имена.

**Результат:**
- VM1: `enp0s1` = внешний (192.168.188.3), `enp0s2` = внутренний
- VM2: `enp0s1` = внутренний
- VM3: `enp0s1` = внутренний

### Шаг 2 — Включение systemd-networkd (на всех VM)

```bash
sudo systemctl enable systemd-networkd
sudo systemctl start systemd-networkd
sudo systemctl enable systemd-resolved
sudo systemctl start systemd-resolved
sudo rm -f /etc/resolv.conf
sudo ln -sf /run/systemd/resolve/stub-resolv.conf /etc/resolv.conf
```

**Зачем:** В Debian 12 пакет `ifupdown` не установлен по умолчанию. Файл `/etc/network/interfaces` не работает. Вместо него используется `systemd-networkd` — современная система управления сетью, встроенная в systemd.

`systemd-resolved` — отдельная служба для DNS. Симлинк `/etc/resolv.conf → stub-resolv.conf` нужен чтобы приложения находили DNS через resolved.

### Шаг 3 — Настройка статических IP через systemd-networkd

Конфигурационные файлы создаются в `/etc/systemd/network/` с расширением `.network`.

**VM1 — внешний интерфейс (enp0s1):**
```bash
sudo nano /etc/systemd/network/10-enp0s1.network
```
```ini
[Match]
Name=enp0s1

[Network]
Address=192.168.188.3/24
Gateway=192.168.188.1
DNS=8.8.8.8
DNS=8.8.4.4
```

**VM1 — внутренний интерфейс (enp0s2):**
```bash
sudo nano /etc/systemd/network/20-enp0s2.network
```
```ini
[Match]
Name=enp0s2

[Network]
Address=192.168.100.1/24
```

**VM2:**
```bash
sudo nano /etc/systemd/network/10-enp0s1.network
```
```ini

[Network]
```

**VM3:**
```ini

[Network]
```

**Применение на каждой VM:**

**Зачем:** статические IP нужны чтобы адреса не менялись при перезагрузке. У VM2 и VM3 были одинаковые IP от UTM DHCP — статика решила эту проблему.

### Шаг 4 — IP-форвардинг на VM1

```bash
# Временно (проверка)
sudo sysctl -w net.ipv4.ip_forward=1

# Постоянно
echo 'net.ipv4.ip_forward=1' | sudo tee -a /etc/sysctl.conf
sudo sysctl -p

# Проверка (должно быть 1)
cat /proc/sys/net/ipv4/ip_forward
```

**Зачем:** по умолчанию Linux не пересылает пакеты между интерфейсами — он принимает только пакеты адресованные себе. IP-форвардинг говорит ядру: «если пришёл пакет не для меня, но я знаю куда его отправить — перешли его дальше». Без этого VM2/VM3 могут пинговать VM1, но через неё в интернет не выйдут.

### Шаг 5 — NAT (MASQUERADE) через iptables на VM1

```bash
# Очистка старых правил
sudo iptables -F
sudo iptables -t nat -F

# Включение маскарадинга
sudo iptables -t nat -A POSTROUTING -o enp0s1 -j MASQUERADE

# Разрешение форвардинга
sudo iptables -A FORWARD -i enp0s2 -o enp0s1 -j ACCEPT
sudo iptables -A FORWARD -i enp0s1 -o enp0s2 -m state --state ESTABLISHED,RELATED -j ACCEPT

# Проверка
sudo iptables -t nat -L -v -n
```

**Зачем:** VM2 и VM3 имеют «серые» (приватные) IP-адреса `192.168.100.x`. Интернет не знает как отвечать на эти адреса. MASQUERADE подменяет адрес отправителя: когда VM2 шлёт пакет на 8.8.8.8, VM1 меняет адрес источника с `192.168.100.10` на свой `192.168.188.3`. Ответ приходит на VM1, которая знает что его надо вернуть VM2.

Это называется **NAPT (Network Address and Port Translation)** или маскарадинг.

### Шаг 6 — Firewall (iptables) на VM1

```bash
# Разрешающие правила (СНАЧАЛА!)
sudo iptables -A INPUT -i lo -j ACCEPT
sudo iptables -A INPUT -m state --state ESTABLISHED,RELATED -j ACCEPT
sudo iptables -A INPUT -p icmp -j ACCEPT
sudo iptables -A INPUT -p tcp --dport 22 -s 192.168.100.0/24 -j ACCEPT
sudo iptables -A INPUT -p tcp --dport 22 -s 192.168.188.0/24 -j ACCEPT

# Политики DROP (ПОСЛЕ разрешающих!)
sudo iptables -P INPUT DROP
sudo iptables -P FORWARD DROP
sudo iptables -P OUTPUT ACCEPT

# Forwarding правила
sudo iptables -A FORWARD -i enp0s2 -o enp0s1 -j ACCEPT
sudo iptables -A FORWARD -i enp0s1 -o enp0s2 -m state --state ESTABLISHED,RELATED -j ACCEPT

# Проверка
sudo iptables -L -v -n
```

**Зачем каждое правило:**

| Правило | Зачем |
|---------|-------|
| `-i lo -j ACCEPT` | Разрешить loopback — многие сервисы общаются через 127.0.0.1 |
| `ESTABLISHED,RELATED` | Разрешить ответы на уже начатые соединения |
| `icmp -j ACCEPT` | Разрешить ping — нужен для диагностики |
| `dport 22 -s 192.168.100.0/24` | SSH только из внутренней сети |
| `-P INPUT DROP` | Запретить всё входящее что не разрешено явно |
| `-P FORWARD DROP` | Запретить весь транзитный трафик что не разрешён явно |
| `enp0s2 → enp0s1 ACCEPT` | Разрешить клиентам выходить в интернет |
| `enp0s1 → enp0s2 ESTABLISHED` | Разрешить ответы из интернета клиентам |

**Принцип:** «запретить всё, разрешить необходимое» — это основа сетевой безопасности.

## Часть 3. Теоретический минимум

### Сетевая модель

**IP-адрес** — уникальный адрес устройства в сети. Состоит из адреса сети и адреса хоста.

**Маска подсети** — определяет какая часть IP-адреса это сеть, а какая хост. `/24` = `255.255.255.0` = первые 24 бита это сеть.

**Шлюз по умолчанию (default gateway)** — IP-адрес роутера, на который отправляются все пакеты не для локальной сети. Для VM2/VM3 шлюз = VM1 (`192.168.100.1`).

**DNS (Domain Name System)** — переводит доменные имена (`google.com`) в IP-адреса. Без DNS пинг по имени не работает, только по IP.

---

### Маршрутизация

**Маршрутизация** — процесс выбора пути для пакета. Роутер смотрит в таблицу маршрутизации и решает через какой интерфейс отправить пакет.

**IP-форвардинг** — разрешение ядру Linux пересылать пакеты между интерфейсами. Без него Linux работает только как конечный хост, не как роутер.

Таблица маршрутизации VM2:
```
default via 192.168.100.1 dev enp0s1   ← всё неизвестное → на VM1
192.168.100.0/24 dev enp0s1            ← локальная сеть напрямую
```

---

### NAT и MASQUERADE

**NAT (Network Address Translation)** — подмена IP-адресов в заголовках пакетов.

**NAPT / MASQUERADE** — частный случай NAT: подменяет адрес источника на адрес исходящего интерфейса. Позволяет множеству устройств с приватными IP выходить в интернет через один публичный IP.

Как работает:
1. VM2 (`192.168.100.10`) шлёт пакет на `8.8.8.8`
2. VM1 меняет источник: `192.168.100.10` → `192.168.188.3`, запоминает это в таблице
3. `8.8.8.8` отвечает на `192.168.188.3`
4. VM1 смотрит таблицу, меняет обратно: `192.168.188.3` → `192.168.100.10`
5. VM2 получает ответ

---

### iptables

**iptables** — утилита для управления netfilter (встроенный в ядро Linux файрволл).

**Таблицы:**
- `filter` — основная, фильтрация пакетов (цепочки INPUT, OUTPUT, FORWARD)
- `nat` — трансляция адресов (цепочки PREROUTING, POSTROUTING)

**Цепочки:**
- `INPUT` — пакеты для самого хоста
- `OUTPUT` — пакеты от самого хоста
- `FORWARD` — пакеты проходящие через хост (транзитные)
- `POSTROUTING` — пакеты после принятия решения о маршруте (здесь делается MASQUERADE)

**Политики:**
- `ACCEPT` — разрешить (по умолчанию)
- `DROP` — отбросить без уведомления отправителя
- `REJECT` — отбросить с уведомлением

**Правило `-m state --state ESTABLISHED,RELATED`** — разрешает пакеты принадлежащие уже установленным соединениям. Без этого правила при политике DROP ответы из интернета не пройдут обратно.

---

### systemd-networkd vs ifupdown

| | ifupdown (старый) | systemd-networkd (новый) |
|--|--|--|
| Конфиг | `/etc/network/interfaces` | `/etc/systemd/network/*.network` |
| Команды | `ifup`, `ifdown` | `networkctl` |
| Debian 12 | не установлен по умолчанию | используется по умолчанию |
| DNS | через `/etc/resolv.conf` | через `systemd-resolved` |

## Часть 4. Вопросы к сдаче

### Базовые вопросы

**Q: Зачем нужен IP-форвардинг на VM1?**
A: По умолчанию Linux принимает только пакеты адресованные ему. IP-форвардинг разрешает пересылать пакеты между интерфейсами — без него VM1 не может работать как маршрутизатор.

**Q: Почему MASQUERADE применяется на enp0s1, а не на enp0s2?**
A: MASQUERADE подменяет адрес источника на адрес исходящего интерфейса. Пакеты выходят в интернет через enp0s1, поэтому маскарадинг применяется там. На enp0s2 это не имеет смысла — через него трафик идёт во внутреннюю сеть.

**Q: Что делает правило `-m state --state ESTABLISHED,RELATED`?**
A: Разрешает пакеты принадлежащие уже установленным соединениям и связанным с ними. Например, когда VM2 отправила запрос в интернет — ответ нужно пропустить обратно. Без этого правила при политике DROP ответы будут заблокированы.

**Q: Почему политика INPUT установлена в DROP, а OUTPUT в ACCEPT?**
A: INPUT DROP — принцип «запретить всё входящее кроме явно разрешённого». Защищает от несанкционированных подключений извне. OUTPUT ACCEPT — VM1 сама инициирует только нужные соединения, ограничивать исходящий трафик нет смысла в данной лабораторной.

**Q: Почему у VM2 и VM3 одинаковый шлюз?**
A: Оба клиента выходят в интернет через VM1. VM1 имеет адрес `192.168.100.1` во внутренней сети — это и есть их шлюз.

**Q: Что произойдёт если на VM2 указать шлюз 192.168.188.1?**
A: VM2 попытается отправить пакеты напрямую на `192.168.188.1` (домашний роутер), но этот адрес недоступен из сети `192.168.100.0/24`. Интернет работать не будет.

**Q: Почему в Debian 12 не используется ifupdown?**
A: Разработчики Debian перешли на `systemd-networkd` как более современную и интегрированную систему. ifupdown устарел и не устанавливается по умолчанию начиная с Debian 12.

**Q: В чём отличие systemd-networkd от NetworkManager?**
A: systemd-networkd — лёгкая серверная система, работает без GUI, идеальна для серверов. NetworkManager — для десктопов, поддерживает Wi-Fi, VPN, имеет GUI. На серверах используют networkd, на десктопах — NM.

**Q: Объясните разницу между маршрутизацией и NAT.**
A: Маршрутизация — решение куда отправить пакет (через какой интерфейс, к какому следующему хопу). NAT — подмена адресов в пакете. Маршрутизация работает на уровне пути, NAT — на уровне адресов. Оба нужны для выхода клиентов в интернет: маршрутизация определяет путь, NAT позволяет использовать приватные адреса.

**Q: Что такое default gateway?**
A: Маршрут по умолчанию — адрес роутера куда отправляются все пакеты для которых нет более конкретного маршрута в таблице маршрутизации.

**Q: Как работает MASQUERADE?**
A: При выходе пакета через интерфейс подменяет адрес источника на IP этого интерфейса, запоминает соответствие в таблице conntrack. При получении ответа восстанавливает оригинальный адрес.

### Вопросы по диагностике

**Q: Как проверить работает ли IP-форвардинг?**
```bash
cat /proc/sys/net/ipv4/ip_forward
# 1 = включён, 0 = выключен
```

**Q: Как посмотреть текущие правила NAT?**
```bash
sudo iptables -t nat -L -v -n
```

**Q: Пинг по IP работает, но по имени нет — в чём проблема?**
A: Проблема с DNS. Проверить: `nslookup google.com`, `cat /etc/resolv.conf`, `systemctl status systemd-resolved`.

**Q: VM2 пингует VM1, но не пингует 8.8.8.8 — где проблема?**
A: Либо не включён IP-форвардинг на VM1, либо не настроен NAT. Проверить: `cat /proc/sys/net/ipv4/ip_forward` и `sudo iptables -t nat -L -v -n`.

**Q: Как убедиться что пакеты доходят до VM1 от VM2?**
```bash
sudo tcpdump -i enp0s2 -n icmp
# Запустить на VM1 пока VM2 пингует
```
